# セクション1: 自律型AIエージェントの全体像

## はじめに
この章では、昨今のAI分野で最も注目されている「自律型AIエージェント」について、専門的な知識を持たない方にも分かりやすく解説します。

「そもそもAIエージェントとは何か」という基本概念から始まり、その裏側で動いている仕組み、マニュアル通りに動く従来のプログラムとの違い、そして複数のAIがチームを組む最新の設計パターンまで、直感的な比喩や具体的なタスクの流れを交えながら全体像を掴んでいきましょう。

---

## 1. AIエージェントとは何か？（概念と定義）

これまで私たちが使ってきたChatGPTのようなチャット型AIは、人間が質問や指示（プロンプト）を入力し、それに答えるだけの「優秀な辞書やアドバイザー」のような存在でした。

一方、**AIエージェント（または自律型AIエージェント）**は、「自ら計画を立てて動く、優秀なアシスタント」です。
人間から「この目標を達成してほしい」とゴールを与えられると、自分で「これを達成するには、まずAを調べて、次にBを実行すればよさそうだ」と**計画**を立て、必要に応じて自ら**道具（ツール）**を使いこなしながら、自律的に作業を進めてくれます。

**💡 身近な比喩：ロボット掃除機**
> 単なるAIが「ゴミはここにあります」と教えてくれるだけの存在だとすれば、AIエージェントは「部屋の形やゴミの場所を自分で判断し、壁にぶつかったらルートを変更して、最終的に部屋全体を綺麗にするロボット掃除機」のような存在です。

### 1-1. 「生成AI」との違い

AIエージェントを理解するうえで、まず「生成AI（Generative AI）」との違いを押さえておくことが重要です。

**生成AI（例：ChatGPT）**とは、テキスト、画像、コードなどのコンテンツを「生成する能力」に特化したAIモデルです。人間の問いかけ（プロンプト）に対して1回の推論で回答を返す「一問一答」の形が基本であり、会話が終わると次の指示を待ちます。みずから外部ツールを実行したり、次のステップを計画したりすることはありません。

**AIエージェント**は、この生成AIを「エンジン（思考の核）」として内部に搭載しつつ、さらに以下の能力を上乗せしたシステムです。

| 比較軸 | 生成AI（例：ChatGPT単体） | AIエージェント（例：Claude Code） |
|---|---|---|
| **動き方** | 一問一答（単発応答） | 目標に向けて複数ステップを自律実行するループ |
| **ツール利用** | なし（生成のみ） | あり（Web検索、ファイル操作、コマンド実行など） |
| **計画立案** | なし（人間が都度指示） | あり（LLMが自らサブタスクを分解して計画する） |
| **記憶・文脈** | 会話コンテキスト内のみ | 外部メモリやファイルに情報を永続保存できる |
| **自律度** | 低い（人間が主導） | 高い（目標さえ与えれば自律的に完遂を目指す） |
| **失敗への対処** | 人間が検出して修正する | エラーを自ら検知し、リトライ・修正・報告を実行する |

> **💡 まとめると**：生成AIが「優れた助言をくれる知識人」だとすれば、AIエージェントは「その知識人が自分の手と足をもって、依頼された仕事を最後まで遂行してくれる優秀なアシスタント」です。

## 2. 自律型AIエージェントの仕組み

AIエージェントは魔法のように動いているわけではなく、内部にいくつかの役割を持った「部品」を持っています。主な部品とその働きを見てみましょう。

### 2-1. 主なパーツ（コンポーネント）
- **Planner（プランナー）**: 目標を達成するための手順を細かく分解して考える「脳」の役割です。
- **Executor（実行役）**: プランナーが考えた手順を実際に実行する「手足」の役割です。
- **Tools（ツール・道具）**: Web検索、ファイルの読み書き、計算機など、実行役が外部とやり取りするために使う「道具」です。
- **Memory / Rules（記憶とルール）**: プロジェクトのルールや、「さっきどこまで作業したか」といった履歴を保持する「記憶領域」です。
- **Guardrails（ガードレール）**: 重要なファイルを勝手に消去しないようブロックするなど、危険な操作を防ぐ「安全装置」です。
- **Evals（評価機能）**: 出てきた結果が品質基準を満たしているかを採点・チェックする仕組みです。
- **Human-in-the-loop（人間の介入）**: どうしても自力で判断できない時や、最終確認の際に「人間の承認」を求める仕組みです。

### 2-2. 基本的な働き方（実行ループ）
エージェントは、基本的に以下のループを繰り返して作業を進めます。

1. **Plan（計画）**: ゴールに向けて、次に何をすべきか考える。
2. **Act（実行）**: ツールを使って実際に行動する。
3. **Observe（観察）**: 行動した結果どうなったか（成功したか、エラーが出たか）を確認する。
4. **Reflect（反省・修正）**: 観察結果を踏まえ、必要なら計画を修正する。

このループを、目標が達成されるか、もしくは危険を察知して安全に停止するまで繰り返します。

## 3. 「ワークフロー型」と「自律型」の違い

AIに仕事を任せる方法には、大きく分けて「ワークフロー型」と「自律型」の2つのパターンがあります。

### 3-1. ワークフロー型エージェント
人間があらかじめ手順（ステップ1、ステップ2…）をしっかり決めておき、AIはその「決められた手順の枠内」で作業を行う方式です。
- **💡 比喩**: 敷かれたレールの上を時刻表通りに走る「電車」や、マニュアル通りに動く「工場ライン」。
- **メリット**: 予想外のミスが起きにくく、管理が簡単で安全です。
- **デメリット**: マニュアルにない想定外の事態（例外）が起こると、対応できず止まってしまいます。

### 3-2. 自律型エージェント
目的地（ゴール）だけを伝えて、達成までの道のりはAI自身に都度判断させる方式です。
- **💡 比喩**: 目的地を入力すれば、渋滞や工事を避けてルートを自動で変更しながら進む「自動運転車」。
- **メリット**: 人間がマニュアル化しきれない複雑な作業や、探索が必要な課題に強いです。
- **デメリット**: 思いがけない行動に出たり、途中で「迷子」になったりするリスクがあるため、前述の「ガードレール（安全装置）」の設定が非常に重要になります。

**【実務での使い分けルール】**
最初からすべてを自律型に任せるのはリスクを伴います。まずは手順が明確な**ワークフロー型**から始め、手順が複雑で予測しきれない部分に絞って**自律型**を取り入れていくのが安全かつ確実な進め方です。

## 4. 自律型AIエージェントの基本パターンから「マルチエージェント」へ

AIエージェントが複雑な課題に挑む際、どのように思考し、仕事を進めていくかという「設計パターン（定石）」がいくつか存在します。ここでは基本的なシングルエージェントのパターンから順に紹介し、最後に応用形である「マルチエージェント」へと繋げます。

各パターンには「概要・比喩」「具体的なタスクの流れ」「メリット・デメリット」「特に有効なケース」の4点を記載しています。

### 4-1. 基本的な設計パターン（シングルエージェントの定石）

---

**1. Prompt Chaining（直列リレー）**

- **概要**: 複雑なタスクを細かいステップに分け、前の工程の結果を次の工程の入力として順番に処理する方式です。
- **💡 比喩**: 工場のベルトコンベアや、書類のバケツリレー。
- **具体的なタスクの流れ（例: ブログ記事の作成）**:
  1. 【人間】「長時間の会議録をブログ記事にして」と依頼する。
  2. 【AI第一工程】会議録テキストから「重要なトピック」だけを箇条書きで抽出する。
  3. 【AI第二工程】その箇条書きを受け取り、「ブログの構成（見出しと骨組み）」を作成する。
  4. 【AI第三工程】構成を受け取り、「読みやすい文章」へと清書して人間に出力する。
- **メリット**: 工程ごとにAIの専門性を分けられるため、各ステップの精度が上がります。また各工程を個別にテスト・修正しやすく、品質管理がしやすいのが強みです。
- **⚠️ デメリット**: ステップ数が増えるにつれ、処理時間とコストが線形に増加します。また前の工程でミスが発生すると、そのエラーが後続の工程全体に波及する「連鎖エラー」のリスクがあります。
- **✅ 特に有効なケース**: 「文書の要約 → 翻訳 → レイアウト整形」のように、処理の手順と品質基準が明確に定まっている定型業務に最も適しています。手順があいまいな探索的なタスクには向きません。

---

**2. Routing（振り分け・受付窓口）**

- **概要**: 入力内容を分類し、その目的に最適な専門処理ツール（または専用のプロンプト）にタスクを割り振る方式です。
- **💡 比喩**: 総合病院の「総合受付（まずは症状を聞いて、内科や外科に案内する）」。
- **具体的なタスクの流れ（例: お問い合わせ対応）**:
  1. 【人間】チャットボットに「パスワードを忘れた」と入力する。
  2. 【AI受付窓口】入力された文章を分析し、「これは『アカウント支援』のカテゴリーである」と判定する。
  3. 【AIアカウント処理係】受付から引き継いだ判定を元に、データベースを検索してパスワードリセットの手順をユーザーに回答する。
- **メリット**: 単一の巨大なAIにすべてを任せるよりも、分野ごとに最適化された専門AIへ誘導することで、全体の回答品質と処理速度が向上します。
- **⚠️ デメリット**: 振り分け（分類）の精度が全体の品質のボトルネックになります。受付AIが入力内容を誤分類した場合、誤った専門家に引き継がれてしまい、的外れな回答が生まれます。
- **✅ 特に有効なケース**: 問い合わせの種別（返品、支払い、技術サポートなど）があらかじめ明確に定義できるカスタマーサービス、あるいは複数の言語や形式のコンテンツを種別ごとに異なる処理で扱う必要がある場面。

---

**3. Evaluator-Optimizer（評価と改善の反復）**

- **概要**: 「作成担当」と「評価担当」のロジックを別々に用意し、厳しい基準を満たすまで何度も作り直しとチェックを繰り返す方式です。
- **💡 比喩**: 原稿を書く「新人作家」と、それにダメ出しをする「厳しい編集者」。
- **具体的なタスクの流れ（例: プログラムコードの生成）**:
  1. 【AI作成担当】指定された要件に従ってプログラムのコードを書く。
  2. 【AI評価担当】そのコードを検査し、「この行でエラーが出る」「セキュリティが甘い」と具体的なダメ出しリストを作成して差し戻す。
  3. 【AI作成担当】ダメ出しリストを元にコードを修正し、再提出する。（評価担当が「OK」を出すまで繰り返す）。
- **メリット**: 反復を重ねるほどアウトプットの品質が向上するため、「一発回答」では難しい高品質なコード・文章・計画書の生成に非常に強いです。
- **⚠️ デメリット**: 評価基準（何をもって「OK」とするか）が曖昧だと、反復が際限なく続く「無限ループ」に陥るリスクがあります。反復のたびにAPIコストも発生するため、収束しないと費用が膨らみます。
- **✅ 特に有効なケース**: テストが自動化されているコード生成、特定の公式基準に適合しなければならない法的文書や技術仕様書のドラフト作成など、「合格・不合格」が明確に判定できるタスク全般。

---

**4. ReAct（思考と行動のループ）**

- **概要**: 「考える(Thought)」→「行動(Action)」→「結果の観察(Observation)」のサイクルを回し、環境の反応に合わせて計画をその都度調整していく方式です。
- **💡 比喩**: 知らない街で「地図を見る」→「少し歩く」→「看板を確認する」を繰り返して目的地に辿り着く人。
- **具体的なタスクの流れ（例: 最新情報の検索回答）**:
  1. 【人間】「今日の日経平均株価を教えて」と質問する。
  2. 【AI・思考】「自分は今の株価の記憶がないから、Web検索ツールを使う必要があるな」
  3. 【AI・行動】Web検索ツールに「日経平均 本日」と入力して実行する。
  4. 【AI・観察】返ってきた検索結果の文章を読み、「最新の株価は〇〇円だ」という事実を見つける。
  5. 【AI・行動】発見した事実を人間に分かりやすい文章にまとめて回答する。
- **メリット**: 外部環境（Web、データベース、ファイルシステムなど）と動的にやり取りしながら情報を収集できます。AIが持つ「学習済み知識の限界（情報の古さ）」をリアルタイムの情報取得で補えます。
- **⚠️ デメリット**: 外部ツールから返ってくる情報（観察結果）にノイズや誤情報が含まれると、AIがそれを正しいものとして取り込み、誤った方向へ推論を強化してしまうリスクがあります（ハルシネーションの増幅）。
- **✅ 特に有効なケース**: 検索エンジン・API・ファイルなど外部のツールを駆使してリアルタイム情報に基づく回答を生成する「リサーチエージェント」、複数ソースを参照しながら調査・デバッグが必要な「トラブルシューティングエージェント」全般。

---

**5. Reflexion（反省と記憶）**

- **概要**: 一度失敗した行動や結果を「反省メモ」として自ら言語化・記憶し、次の挑戦で同じ間違いをしないように学習（適応）する仕組みです。
- **💡 比喩**: スポーツ選手が試合後にビデオを見て「次はここを直そう」と自己分析ノートに書き留めること。
- **具体的なタスクの流れ（例: スクリプトの自動実行と修正）**:
  1. 【AI】システムを操作するスクリプトを書いて実行するが、エラーで失敗する。
  2. 【AI反省プロセス】エラーメッセージを読み込み、「なぜ失敗したか（例：変数のスペルを間違えていた）」を分析し、反省テキストとしてファイルに保存する。
  3. 【AI】次回の再試行時、自身の反省テキストを読み込み、「同じ変数名の間違いをしないぞ」と意識しながら再プログラミングして成功させる。
- **メリット**: モデルの重みを再学習させることなく、実行のたびに行動が改善されます（ファインチューニング不要のオンライン改善）。失敗を単なるエラーとして捨てるのではなく、次の成功への「学習リソース」に変換できます。
- **⚠️ デメリット**: 反省メモのクオリティが低い（「なぜ失敗したかが的外れ」）と、誤った学習が蓄積されてかえって性能が低下するリスクがあります。また記憶の量が増えすぎると、コンテキスト長（AIが一度に処理できる情報量）を圧迫する可能性があります。
- **✅ 特に有効なケース**: 何度も繰り返しトライして少しずつ改善できる試行錯誤型のタスク全般。プログラムのデバッグループや、ゲームのクリアを目指す行動パターンの最適化など。

---

**6. Tree of Thoughts / ToT（思考の枝分かれ探索）**

- **概要**: 一本道ではなく、複数の選択肢（思考の枝）を同時に検討し、「この道は行き止まりだ」と判断したら別の枝を探す、将棋の先読みのような方式です。
- **💡 比喩**: 複雑な迷路で「右に行ったらどうなるか」「左に行ったらどうなるか」を頭の中で数手先までシミュレーションして正解ルートを探すこと。
- **具体的なタスクの流れ（例: 難解な論理パズルの解答）**:
  1. 【AI】パズルを解く際、「まずは足し算から始めるルート案」と「引き算から始めるルート案」の2つの手順を同時に生成する。
  2. 【AI】それぞれのルートの中間結果を評価し、「足し算ルートはすぐに矛盾が生じる。引き算ルートなら次のステップに進めそうだ」と見極める。
  3. 【AI】見込みのない足し算ルートの計算を打ち切り、引き算ルートに計算リソースを集中して最短で正解を導き出す。
- **メリット**: 一発で解くのが難しい複雑な探索・計画問題において、見込みのない選択肢を早期に切り捨てることで無駄な計算を省き、最適解に効率よくたどり着けます。
- **⚠️ デメリット**: 探索する「枝」の幅と深さを広げすぎると、計算量が指数関数的に増加し、非常にコストと時間がかかります。探索空間の設計（どこまで先読みするか）が難しく、過剰に広い探索は実用的ではありませんん。
- **✅ 特に有効なケース**: 数学的な証明問題、コードの複数ある実装方針の選定、複数の選択肢を検討しなければならない中長期的な戦略立案など、「答えが一つで、かつ選択肢が複数存在する問題」。

---

**7. Plan-and-Execute（計画先行・分離実行）**

- **概要**: まず全体の実行計画（タスクリスト）を一括で立案し、その後にプランに沿って各ステップを順次・あるいは並列に実行する方式です。「先に地図を作ってから出発する」ことで、ReActのような都度判断ではなく、より見通しの良い作業進行を実現します。
- **💡 比喩**: 旅行の前に「行程表」を完全に作り上げて、あとはその通りに動く旅行プランナー。一方ReActは「とりあえず出発して、その場その場で次どこに行くか決める旅人」です。
- **具体的なタスクの流れ（例: 大規模なリファクタリング依頼）**:
  1. 【人間】「このコードベース全体をリファクタリングして」と依頼する。
  2. 【Planner AI（計画フェーズ）】コードベースを分析し、「① 不要なimportの削除」「② 関数の分割」「③ テストの追加」「④ ドキュメントの更新」という4ステップの具体的な実行計画リストを生成する。
  3. 【Executor AI（実行フェーズ）】作成された計画リストを受け取り、①→②→③→④ の順（あるいは③と④を並列）に実行していく。
  4. 各ステップの完了後、Plannerが当初の計画と照らし合わせて差異がないかをチェックする。
- **メリット**: 全体像を先に把握してから動くため、途中で方向性がブレにくく、大規模・長時間のタスクに向いています。計画段階で「何をする必要があるか」を人間がレビューしやすく、承認後に実行するフローを組み込みやすい点も利点です。
- **⚠️ デメリット**: 計画立案フェーズで状況を見誤ると、その後の全工程が誤った方向に進んでしまいます（計画の品質がすべての起点）。また途中で状況が変わった場合（例: コードの前提が変わった）、事前計画を修正することが難しく、柔軟性の面ではReActに劣ります。
- **✅ 特に有効なケース**: 「タスクの全体量が事前にある程度見積もれる大規模作業」や「工程を人間が事前確認してから実行したい場合」。CursorのBackground Agent、Windsurf（Cascade）、OpenAI Codexなど、主要なAIコーディングエージェントの多くが採用している設計パターンです。

---

### 4-2. 複数のAIで挑むアプローチ

基本的なパターンを組み合わせ、複数の処理を並行させたり、完全に役割分担させたりする発展形へのアプローチです。

---

**8. Parallelization（並列処理・多数決）**

- **概要**: 複数のエージェント（または別々のプロセス）に同じ課題や分割した課題を同時に解かせ、その結果を統合したり多数決で決めたりする方式です。
- **💡 比喩**: 企画会議での「ブレインストーミング（アイデア出し）」や「多数決による決議」。
- **具体的なタスクの流れ（例: キャッチコピーの作成）**:
  1. 【人間】「新機能のキャッチコピーを考えて」と依頼する。
  2. 【AI並列処理】AIが同時に3つの異なるプロンプト（「ユーモア重視」「真面目なトーン重視」「短いインパクト重視」）を起動し、3バージョンのコピー案を同時に書き上げる。
  3. 【AI統合処理】出来上がった3つの全く異なる案を別の評価プロセス（または人間）が集め、「一番目的に合ったもの」を選ぶ。
- **メリット**: 複数の視点から同時に問題に取り組むため、単一の処理では見落としやすい解法や切り口を発見できます。また直列処理に比べて、タスクを並行実行することで全体の処理時間を短縮できます。
- **⚠️ デメリット**: 複数の結果を最終的にひとつにまとめる「統合・集約ロジック」の品質が弱いと、「速いが一貫性のない粗い成果物」になりやすいです。また並列実行する分、APIの呼び出し回数が増えるためコストが倍増します。
- **✅ 特に有効なケース**: 「複数の異なる視点の意見が欲しいとき（多様なアイデア出し）」、または「長い文書を章ごとに分割してそれぞれ独立して処理したいとき（分割統治）」。どちらの場合も、最終的な集約ロジックの設計が成功の鍵です。

---

**9. Orchestrator-Workers（オーケストレーション）**

- **概要**: これまでの要素の集大成です。リーダー（オーケストレーター）が全体の大きな目標を複数の小さなタスクに分解し、それぞれ得意分野を持つ「専門家エージェントチーム（ワーカー）」に指示を出して、最後に一つの成果物にまとめます。
- **💡 比喩**: 建築現場を仕切る「プロジェクトマネージャー（現場監督）」と、その指示で動く大工・配管工・電気工事士などの「専門スタッフたち」によるチームプレイ。
- **具体的なタスクの流れ（例: 大規模なソフトウェア開発）**:
  1. 【人間】「この既存アプリに、ログイン認証機能を追加して」という大雑把で複雑な依頼を出す。
  2. 【オーケストレーターAI】依頼を解釈し、「1. 既存構造の調査」「2. バックエンドの実装」「3. フロントエンドの実装」という3つのタスクが必要だと判断する。
  3. 【オーケストレーターAI】調査担当ワーカー、サーバー側担当ワーカー、画面側担当ワーカーの3体にそれぞれ仕事と必要な権限を割り当てる。
  4. 【各ワーカーAI】並行して自分の持ち場のコードを読み書きする。
  5. 【オーケストレーターAI】各ワーカーの報告を回収し、すべてのコードが矛盾なく繋がっているかを統合確認して人間に納品する。
- **メリット**: 1体のAIでは処理しきれない大規模・複雑なタスクを、役割分担によって現実的な時間内で遂行できます。各ワーカーが専門領域に特化できるため、全体のアウトプット品質が大幅に向上します。
- **⚠️ デメリット**: オーケストレーターが行う「タスクの分解」の品質が低いと、誤った方向で全ワーカーが同時に動いてしまいます（親の失敗が全員に波及）。システム全体の設計・評価・監視が複雑になるため、導入コストが高いです。
- **✅ 特に有効なケース**: 「大規模なコードベースのリファクタリング」「複合的なリサーチレポートの自動生成」のように、タスクを事前に完全に列挙できないほど複雑で、かつ各工程でそれぞれ異なる専門知識が求められる長大な業務。

### 4-3. 【重要】「ワークフロー」と「AIエージェント」の境界線と具体名称

マルチエージェントの「オーケストレーション」パターンを採用していても、その中身によって「単なるワークフロー」か「真のAIエージェント」かが分かれます。この境界線は**「タスクの分割と割り振りを、誰が（どう）決めているか」**にあります。代表的な具体名称とともに見てみましょう。

**1. 静的なオーケストレーション（例：Router Pattern / Workflow Management）**
> 「もし条件Aなら文章担当ワーカーへ、条件Bならコード解析ワーカーへ」と、人間があらかじめプログラムでルール（if-thenの分岐など）を固定している状態です。タスクの振り分け方は事前に定義されており、これはマニュアル通りの**単なるワークフロー（Workflow）**に属します。

**2. 動的なオーケストレーション（例：Supervisor Pattern / Hierarchical Agents）**
> リーダー（Supervisorと呼ばれる監督エージェントなど）自身がLLMの高度な推論能力を使います。「今回のユーザーの目標を達成するには、タスクを3つに分解して、それぞれリサーチ、執筆、レビューの専門家に任せるのが最適だ」とその場で考え、**動的にタスクを定義・割り振り（Dynamic Task Decomposition）**します。
> このように、あらかじめ敷かれたレールはなく、指揮官自身が状況に合わせて計画を生成し、指示を下す形こそが、高度な**マルチAIエージェント（Agentic Workflow）**の振る舞いと言えます。

## 5. まとめと用語集

### この章の要点
- AIエージェントは、単に質問に答えるだけではなく、自ら計画し、道具を使い、観察してゴールへと向かう**「自律的なアシスタント」**である。
- 最初は安全で確実な**「ワークフロー型（電車）」**を採用し、複雑な課題には経路を自分で探す**「自律型（自動運転車）」**を適材適所で使うのが良い。
- AIの思考回路には、直列リレーや受付窓口、評価と改善のループ、計画先行型の分離実行など、**9つの代表的な設計パターン**が存在し、これらを組み合わせてタスクをこなしている。
- 1つのAIでは解決が難しい大きな課題には、シングルエージェントの定石を応用した**「マルチエージェントオーケストレーション」**が有効である。
- ただし、本物のAIエージェントと呼べるのは、Supervisorパターンのように「AI自らがタスクを動的に分解して割り振る」仕組みを持ったものである。

## 6. 主要AIコーディングエージェントと採用設計パターン

各ツールがどのような設計パターンを組み合わせているかを知ることで、選択・活用の判断軸が生まれます。以下は公開ドキュメント・公式ブログに基づいた一次調査の結果です。

> ⚠️ **調査方針について（deep-research skillによる事実確認済み）**
> - **【確認済み】**: 公式ドキュメント・公式ブログで明示されている事実
> - **【推察】**: 公開情報を元にした合理的な推定（内部実装の詳細は非公開）

### 6-1. 製品別パターンマッピング

> **🔍 補足：ReAct と Plan-and-Execute の違い**
> - **ReAct** = Think（思考）→ Act（行動）→ Observe（観察）を**一手ずつ交互に繰り返す**逐次的なアプローチ。状況が変わるたびに即興で方針を修正します。
> - **Plan-and-Execute** = まず「全体の実行計画（タスクリスト）」を立案し、その後にプランに沿って各ステップを順次・並列に実行するアプローチ。大きく複雑なタスクに向いています。

| ツール | 主要な設計パターン | 根拠・補足 |
|---|---|---|
| **Claude Code** (Anthropic) | **ReAct + Plan-and-Execute のハイブリッド** + Reflexion | 【確認済み】内部に「plan mode（計画モード）」と「execution mode（実行モード）」が存在。通常操作はReActループが中核だが、複雑なタスクでは先に計画を立ててから実行に移る。`CLAUDE.md`は記憶をReflexion的に機能させる |
| **Google Antigravity** | Plan-and-Execute + Orchestrator-Workers（Supervisor型） + Parallelization | 【確認済み】タスク開始時に計画を生成し、専門エージェントチームに分配・並列実行するSupervisor型。Agent Managerが複数エージェントを監視・管理。KnowledgeItem（長期記憶）とArtifactsで状態管理 |
| **Cursor** | Routing + Plan-and-Execute（Background Agent） + Prompt Chaining | 【確認済み】「前景（対話的なPrompt Chaining補完）」と「Background Agent（非同期実行）」を切り替えるRouting構造。Background AgentはCursor 2.0でPlan-and-Execute方式を採用し、最大8つの並列エージェントをサポート |
| **Windsurf** (Codeium) | Plan-and-Execute + Evaluator-Optimizer | 【確認済み】中核機能「Cascade」が「数手先まで考えて計画してから行動する」設計を公式に明記。コード生成→テスト実行→評価→修正のEvaluator-Optimizerループを採用。「エレファントメモリ」と呼ばれる長期コンテキスト保持も特徴 |
| **Codex** (OpenAI) | Plan-and-Execute + Orchestrator-Workers + Parallelization + Reflexion | 【確認済み】エージェントループ内に組み込みの`update_plan`ツールが存在し、Plan-and-Executeを明示的に実装。クラウド上で複数のサンドボックスワーカーが並列実行。`AGENTS.md`による規約はReflexion的な学習支援として機能 |
| **Cline** | ReAct + Routing + Prompt Chaining | 【確認済み】コアに「Reason-Act-Observe（ReAct）ループ」を採用し、ユーザーの承認フローを組み込んだHuman-in-the-loopが特徴。MCP経由でツール接続を切り替えるRoutingも活用 |
| **OpenClaw** | ※コーディングエージェントではないため注記参照 | 【要注意】OpenClawは「WhatsApp/Telegram等と連携するパーソナル汎用エージェント」であり、コーディング特化ツールではありません。コーディングエージェントとして列挙するのは不正確です |

> **📌 OpenClawに関する注記**
> 調査の結果、OpenClawはClaude CodeやCursorのような「コーディング特化エージェント」ではなく、メッセージングプラットフォーム統合型のパーソナルAIアシスタントであることが確認されました。Dockerサンドボックス上で動作し、Skills（スキル）を介してコーディングも一部担えますが、設計の主目的は異なります。もしコーディングエージェントとして比較したい場合、代替候補として **Aider**、**SWE-agent**、**Devin（Cognition Labs）** あたりが適しています。

### 6-2. パターンから見たツール選びの視点

どのツールを選ぶべきか悩んだとき、設計パターンの観点から次のように整理できます。

- **複数のタスクを計画→並行実行させたい** → Plan-and-Execute + Orchestrator-Workers型が強い **Antigravity** や **Codex** が向いています。
- **コードを書いてすぐテスト→修正のサイクルを回したい** → Evaluator-Optimizer型に強みのある **Windsurf** が有効です。
- **IDE内で対話しながら段階的に改修したい** → Routingで前景・背景を使い分ける **Cursor** がバランス良い選択肢です。
- **ターミナルやWebなど多様なツールを横断しつつ都度即興で判断させたい** → ReActによる環境横断が強い **Claude Code** や **Cline** が適しています。
- **人間の確認・承認を細かく挟みながら進めたい** → Human-in-the-loop設計の **Cline** が最も安心感があります。

### 用語集（初学者向けコンパクト版）
初心者が戸惑いがちな用語を、平易な言葉でまとめました。

- **Agentic system（エージェンティック・システム）**: AIが自らツールを使って複数の作業を進める、一連のシステム全体の総称。
- **Sandbox（サンドボックス）**: AIが万が一暴走したり、誤操作をしても深刻な被害（ファイルの全削除など）が出ないよう、あらかじめ用意された隔離された安全な環境（＝砂場）のこと。
- **MCP (Model Context Protocol)**: AIエージェントが、外部のツールやパソコンのファイル、企業のデータベースなどに安全かつ簡単に接続できるようにするための、世界共通の接続ルール（共通語）のこと。
- **Observability（オブザーバビリティ・可観測性）**: 人間が「AIが今どんな手順を踏んで、何を考えているか」を透明化して監視できるようにする仕組みのこと。

ここまでの内容で、AIエージェントが「どのような仕組みで、どのように仕事を進めているか」というイメージが掴めたかと思います。次のセクションでは、実際にこれらの要素をどう実装していくのかについて見ていきましょう。
